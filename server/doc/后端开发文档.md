## 1. 文档范围与产品边界

### 1.1 需求来源

- 功能与权限以仓库内《需求文档》为准。
- 本文档在需求之上约定**工程化与扩展**：微服务形态、etcd 服务发现、热榜（Redis）、Feed 关注时间线；与需求原文若有表述差异，以本节「边界说明」为准。

### 1.2 边界说明（与需求文档的对齐）


| 项     | 说明                                                                   |
| ----- | -------------------------------------------------------------------- |
| Feed  | **仅实现方案 A**：基于用户「关注的贴吧」下的帖子，按发布时间倒序分页，不做热榜穿插、不做推荐算法。                 |
| 热榜    | **毕设扩展**：固定打分公式 + 定时任务扫库（或按时间窗口查询）写入 **Redis**（如 `ZSET`），对外提供热榜查询接口。 |
| 点赞    | **仅新增不取消**：点赞接口幂等，同一用户对同一目标（帖子/评论）最多一条记录。                            |
| 帖子可见性 | **单一可见状态流转**：待审核 / 可见 / 审核拒绝 / 已隐藏。                                  |
| 消息    | **拉取模式**：不做 WebSocket；站内通知存库，客户端轮询或进入页面刷新。                           |
| 搜索    | **简单模糊检索**：SQL `LIKE`，不用 Elasticsearch。                              |
| Redis | **仅用于热榜（及与热榜直接相关的只读缓存键）**，不作为通用业务缓存层。                                |
| 服务发现  | **etcd**：各 Kitex 服务注册到 etcd；客户端通过 Kitex 与 CloudWeGo 生态的 etcd 集成解析实例。 |

## 2. 框架与技术选型


| 类别      | 选型                                 | 用途说明                                |
| ------- | ---------------------------------- | ----------------------------------- |
| 语言      | Go（建议 1.21+，以团队环境为准）               | 全栈统一                                |
| HTTP 网关 | **Hertz**（CloudWeGo）               | 对外 `/api/v1`、鉴权中间件、路由、必要时 BFF 聚合    |
| 内部 RPC  | **Kitex**（CloudWeGo）               | 服务间调用；IDL 驱动代码生成                    |
| IDL     | **Protobuf**                       | 定义 RPC；生成 Kitex 代码                  |
| ORM     | **GORM**                           | 全服务访问 MySQL 的统一方式                   |
| 关系库     | **MySQL 8**                        | 业务主存                                |
| 缓存      | **Redis**                          | 仅热榜等明确约定的场景                         |
| 鉴权      | **JWT**（无状态 Token）                 | 登录签发；网关或业务校验                        |
| 密码存储    | **bcrypt**（或 argon2）               | 注册与改密                               |
| 服务发现    | **etcd** + Kitex Registry/Resolver |                                     |
| 容器编排    | **Docker Compose**（推荐）             | 本地与演示环境一键拉起 MySQL、Redis、etcd、各服务    |
| 日志      | **Hertz / Kitex 自带 logger**        | 不强制引入 Zap；统一字段（如 request_id）在实现阶段约定 |
| 配置      | 环境变量为主，必要时 **YAML**                | 每服务独立；敏感信息不进仓库                      |

## 3. 系统架构

### 3.1 逻辑架构

```
[ Web / 移动端 ]
        |
        v
 [ API Gateway - Hertz ]  <-- 唯一对外 HTTP，前缀 /api/v1
        |
        +--etcd 发现--> [ user-svc (Kitex) ]
        +--etcd 发现--> [ forum-svc (Kitex) ]
        +--etcd 发现--> [ content-svc (Kitex) ]
        +--etcd 发现--> [ relation-svc (Kitex) ]
        +--etcd 发现--> [ notify-svc (Kitex) ]
        +--etcd 发现--> [ admin-svc (Kitex) ]
        +--etcd 发现--> [ rank-svc (Kitex) ]   // 热榜计算任务 + 热榜读接口（或由 gateway 调 rank 只读）
        |
 [ MySQL ]              [ Redis ]  // 热榜
```

- **网关**：只做 HTTP、鉴权、参数校验、路由到下游 RPC；**不写复杂业务 SQL**（除非极小 BFF 聚合）。
- **Feed（方案 A）**：归属建议——**由 content-svc 提供「关注贴吧帖子时间线」RPC**，网关暴露 `GET /api/v1/feed/...`（路径在 API 文档中最终确定）。

### 3.2 服务职责一览


| 服务           | 职责摘要                                           |
| ------------ | ---------------------------------------------- |
| gateway      | 对外 HTTP；JWT 解析；调用各 Kitex Client；统一错误与分页约定      |
| user-svc     | 注册、登录、登出、资料、改密、用户查询                            |
| forum-svc    | 贴吧 CRUD、列表、详情；名称唯一                             |
| content-svc  | 帖子、评论、点赞；贴吧下帖子列表；帖子详情；评论分页；**Feed 时间线（方案 A）**  |
| relation-svc | 关注/取关贴吧与用户；我的关注列表                              |
| notify-svc   | 通知写入与查询、已读；被赞/被评/系统通知（拉取）                      |
| admin-svc    | 举报审计、管理员审帖/审评、删违规、封禁                           |
| rank-svc     | 热榜定时全量/窗口扫描、写 Redis、热榜查询（或读逻辑在 gateway 调 rank） |


服务可合并（例如 admin 并入 content）以降低进程数，但**文档与 IDL 上仍建议按模块边界拆分接口**，便于答辩叙述。

### 3.3 仓库目录结构（单仓 Monorepo，示意）

```
/
├── api/                      # 对外 HTTP 契约（按服务/域拆分的 .md，见第 4 节）
├── idl/                      # Thrift / Proto 定义，生成 Kitex 代码
├── pkg/                      # 公共库：统一错误码、分页、JWT 工具等（无业务 SQL）
├── deploy/
│   └── docker-compose.yml    # etcd、mysql、redis、各服务
├── gateway/                  # Hertz：cmd、biz、router、middleware
├── conf/                         #放config文件，管理mysql，redis，etcd配置等
└── app/
    ├── user/
    ├── forum/
    ├── content/
    ├── relation/
    ├── notify/
    ├── admin/
    └── rank/
        └── cmd / biz / dal …  # 与 CloudWeGo 示例一致的分层
```

| `api/user.md`     | 注册、登录、用户资料等                       |
| `api/forum.md`    | 贴吧                                |
| `api/content.md`  | 帖子、评论、点赞                          |
| `api/relation.md` | 关注                                |
| `api/notify.md`   | 消息列表、已读                           |
| `api/search.md`   | 搜索（若网关聚合自 forum/content，在此写清）     |
| `api/admin.md`    | 举报与管理                             |
| `api/feed.md`     | **仅方案 A**：关注贴吧帖子时间线               |
| `api/rank.md`     | 热榜查询（读 Redis；与 rank-svc 对应）       |


### 4.2 每个 API 文档文件应包含

- 资源前缀与网关路径（均以 `/api/v1` 为根）。
- 是否需要 `Authorization: Bearer <token>`。
- 请求：Method、Path、Query、Body 字段表（类型、必填、示例）。
- 响应：统一结构 + `data` 形状 + 列表类 `total`。
- 典型错误码与业务含义。

### 4.3 统一 HTTP 约定（与需求文档一致）

- 前缀：`/api/v1`。
- 分页：`pageNum`、`pageSize`，默认每页 **10** 条。
- 鉴权：**Token**（JWT），除注册/登录等白名单接口外需登录；管理员接口需角色校验。
- 统一响应体（实现时全网关一致，文档中写死示例）：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

列表建议在 `data` 内包含 `list` 与 `total`（或团队约定的等价结构），**必须在 `api/README.md` 中只定义一种**，全文档沿用。

## 5. 服务发现（etcd + Kitex）

### 5.1 原则

- **Kitex Server**：启动时向 **etcd** 注册服务名、实例地址、元数据（可选权重）。
- **Kitex Client**（部署在 gateway 或跨服务调用方）：通过 **etcd Resolver** 订阅服务名，获取实例列表；负载均衡使用 Kitex 默认或指定策略（如 weighted random）。
- **etcd** 在 Compose 中单节点即可满足毕设；本文档不展开。

### 5.2 约定（实现阶段补全细节）

- 服务注册名：全仓统一命名规范，例如 `tieba.user`、`tieba.forum`（具体格式在实现时写入 `pkg` 常量）。
- 心跳与 TTL：防止僵尸节点；断连后自动摘除 ketix默认配置。
- 本地开发：可直连本机 etcd，或使用 Compose 网络内 `etcd:2379`。

### 5.3 与网关的关系

- **Hertz 仅 HTTP**，通过 **生成的 Kitex Client** 调用下游；**不**让前端直连 Kitex 端口。

### 7.2 用户模块（user-svc）

#### 7.2.1 对外接口范围

- 注册：`POST /api/v1/users/register`
- 登录：`POST /api/v1/users/login`
- 登出：`POST /api/v1/users/logout`
- 当前用户：`GET /api/v1/users/me`
- 他人公开信息：`GET /api/v1/users/{userId}`
- 修改资料：`PUT /api/v1/users/me/profile`
- 修改密码：`PUT /api/v1/users/me/password`

#### 7.2.2 核心表结构（users）


| 字段            | 类型（建议）       | 约束               | 说明                  |
| ------------- | ------------ | ---------------- | ------------------- |
| id            | bigint       | PK               | 用户 ID               |
| account       | varchar(64)  | UNIQUE, NOT NULL | 登录账号（标准化后入库）        |
| password_hash | varchar(255) | NOT NULL         | bcrypt/argon2 哈希    |
| nickname      | varchar(64)  | NOT NULL         | 昵称                  |
| avatar_url    | varchar(255) | NULL             | 头像地址                |
| bio           | varchar(255) | NULL             | 个人简介                |
| role          | varchar(16)  | NOT NULL         | `user` / `admin`    |
| status        | varchar(16)  | NOT NULL         | `normal` / `banned` |
| ban_until     | datetime     | NULL             | 封禁截止时间              |
| created_at    | datetime     | NOT NULL         | 创建时间                |
| updated_at    | datetime     | NOT NULL         | 更新时间                |


索引建议：

- `uk_users_account(account)` 唯一索引
- `idx_users_role_status(role, status)`（管理员筛查可选）

#### 7.2.3 大致实现思路

- **注册**
  - 校验 `account/password` 格式（长度、字符集、密码强度）
  - 若传入头像文件，后端先上传 本地路径，拿到 `avatar_url`
  - 对密码做哈希后写入 `users`（包含 `avatar_url`）
  - 依赖账号唯一索引兜底并发重复注册
- **登录**
  - 按 `account` 查询用户
  - 比对密码哈希，检查 `status` 是否允许登录
  - 签发 JWT（`userId`、`role`、`exp`）
- **登出**
  - 本项目不做 token 黑名单；登出语义为客户端删除 token
- **当前用户**
  - 网关鉴权后，把 `userId` 传给 user-svc 查询返回
- **修改资料**
  - 允许改 `nickname/avatar_url/bio`
  - 若传入头像文件，后端先上传 本地路径，再更新 `avatar_url`
  - 至少一个字段变更才执行更新
- **修改密码**
  - 先校验旧密码，再更新新密码哈希
  - 新旧密码不能相同

#### 7.2.4 约束与边界

- 账号唯一是强约束，依赖数据库唯一索引保证
- 用户与管理员同表，用 `role` 区分
- 认证由网关统一校验，user-svc 聚焦账户数据与密码校验
- 注册接口支持 `multipart/form-data`，头像由后端上传 本地路径 后回写数据库 URL
- 修改资料接口同样支持 `multipart/form-data` 头像上传，流程与注册一致
